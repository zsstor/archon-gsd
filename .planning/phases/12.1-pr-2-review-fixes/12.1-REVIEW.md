## Summary

Reviewed `.archon/lib/migration.sh`, a bash utility library providing migration detection functions for GSD-to-zarchon migration. The script is well-structured with clear documentation and follows defensive bash practices (`set -euo pipefail`).

One critical security issue was identified: the node fallback in `_has_json_field` interpolates variables directly into JavaScript code, creating a command injection vulnerability. This must be fixed before the code is used in production.

## Critical Issues

### CR-01: Command Injection via Node Fallback

**File:** `.archon/lib/migration.sh:31`
**Issue:** The `$file` and `$field` variables are interpolated directly into JavaScript code passed to `node -e`. If either variable contains JavaScript metacharacters (backticks, `$()`, quotes, or semicolons), arbitrary JavaScript code could be executed.

For example, if `file` were set to `'; require('child_process').execSync('malicious-command'); '`, the shell would pass malicious code to node.

While the current callers use controlled inputs, this is a latent vulnerability that could be exploited if the function is reused with untrusted input.

**Fix:**
Pass the file path and field name as arguments to node instead of interpolating them into the script. Use `process.argv` to access them safely:

```bash
# Fallback to node (available in most dev environments)
if command -v node >/dev/null 2>&1; then
  node -e "
    const fs = require('fs');
    const file = process.argv[1];
    const field = process.argv[2];
    const obj = JSON.parse(fs.readFileSync(file, 'utf8'));
    process.exit(obj[field] !== undefined ? 0 : 1);
  " -- "$file" "$field" 2>/dev/null
  return $?
fi
```

This approach passes `$file` and `$field` as command-line arguments to node, which are then accessed via `process.argv[1]` and `process.argv[2]`. The `--` separator ensures the arguments are not interpreted as node flags.

## Info

### IN-01: Redundant Boolean Initialization

**File:** `.archon/lib/migration.sh:86-87`
**Issue:** The variables `has_archon_dir` and `has_version_field` are initialized to `false` but then immediately overwritten by conditional assignments. This is not a bug but adds unnecessary lines.

**Fix:** The current pattern is acceptable for clarity. Alternatively, could be simplified:

```bash
is_partial_migration() {
  local project_root="${1:-.}"
  local has_archon_dir=false
  local has_version_field=false

  [ -d "${project_root}/.archon" ] && has_archon_dir=true

  if [ -f "${project_root}/.planning/config.json" ] && \
     _has_json_field "${project_root}/.planning/config.json" "zarchon_version"; then
    has_version_field=true
  fi
  # ... rest unchanged
}
```

No action required — this is a minor style observation.
