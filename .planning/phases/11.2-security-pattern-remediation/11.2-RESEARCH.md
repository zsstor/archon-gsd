# Phase 11.2: Security Pattern Remediation - Research

**Researched:** 2026-04-18
**Domain:** Shell script security (command injection prevention in bash/Python heredocs)
**Confidence:** HIGH

## Summary

Phase 11.2 addresses the remaining command injection vulnerabilities (CR-02 through CR-05) identified in the 11.1-REVIEW.md code review. These vulnerabilities exist in `logging.sh` functions and the `read_config` fallback in `ai-delegate` where Python heredocs use unquoted delimiters with shell variable interpolation, allowing potential code execution via crafted input.

The remediation pattern is well-established from Phase 11.1: replace unquoted heredocs (`<<EOF`) with quoted heredocs (`<<'EOF'`) and pass data via environment variables rather than shell interpolation. This pattern was successfully applied to 5 similar cases in Phase 11.1, providing a proven template.

**Primary recommendation:** Apply the consistent "env var + quoted heredoc" pattern from Phase 11.1 to all 4 remaining vulnerable functions: `query_history`, `_apply_time_decay`, `_compute_weighted_failure_rate` in logging.sh, and `read_config` in ai-delegate.

## Standard Stack

### Core
| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|
| bash | 5.x | Shell scripting | Standard POSIX shell with heredoc quoting |
| python3 | 3.x | JSON parsing, date handling | Already in use; safer than pure bash for complex parsing |
| os.environ | stdlib | Environment variable access | Standard Python mechanism for shell-to-Python data passing |

### Security Patterns
| Pattern | Description | When to Use |
|---------|-------------|-------------|
| Quoted heredoc | `<<'EOF'` (single quotes around delimiter) | Always when heredoc contains Python/code |
| Env var passing | `VAR="$value" python3 <<'EOF'` | When passing shell data to heredoc code |
| os.environ[] | `os.environ['VAR']` in Python | Reading env vars in heredoc Python code |

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| Python heredoc | Pure bash | More complex, error-prone for JSON/dates |
| Environment variables | Temporary files | More I/O, cleanup needed, race conditions |
| Environment variables | Command-line args | Works for simple cases; env is cleaner for multiple vars |

## Architecture Patterns

### Recommended Secure Heredoc Pattern
```bash
# Pass data via environment variables, use quoted heredoc
VAR1="$value1" VAR2="$value2" python3 <<'EOF'
import os
var1 = os.environ['VAR1']
var2 = os.environ['VAR2']
# ... Python code using var1, var2 ...
EOF
```

### Pattern 1: Environment Variable Passing
**What:** Pass shell variables to Python via environment rather than interpolation
**When to use:** Any time shell data must reach Python heredoc code
**Example:**
```bash
# Source: Phase 11.1 fix in logging.sh:89-95
TASK_TYPE="$task_type" \
MODEL="$model" \
python3 <<'EOF' >> "$log_file" 2>/dev/null || true
import json
import os

entry = {
    "task_type": os.environ['TASK_TYPE'],
    "model": os.environ['MODEL']
}
print(json.dumps(entry))
EOF
```

### Pattern 2: Numeric Validation Before Use
**What:** Validate shell variables are numeric before passing to Python
**When to use:** When Python expects integer input
**Example:**
```bash
# Validate max_age_days is numeric
[[ "$max_age_days" =~ ^[0-9]+$ ]] || max_age_days=30

MAX_AGE_DAYS="$max_age_days" python3 <<'EOF'
import os
max_age = int(os.environ['MAX_AGE_DAYS'])
EOF
```

### Pattern 3: python3 -c to Heredoc Conversion
**What:** Replace inline python3 -c with heredoc for complex code
**When to use:** When python3 -c contains shell variable interpolation
**Example:**
```bash
# Before (vulnerable):
value=$(python3 -c "
import json
with open('$CONFIG_FILE') as f:
    cfg = json.load(f)
")

# After (secure):
value=$(CONFIG_FILE="$CONFIG_FILE" python3 <<'EOF'
import json
import os
with open(os.environ['CONFIG_FILE']) as f:
    cfg = json.load(f)
EOF
)
```

### Anti-Patterns to Avoid
- **Unquoted heredoc delimiter:** `<<EOF` allows shell expansion; always use `<<'EOF'`
- **Triple-quoted interpolation:** `'''$var'''` in Python string is still shell-expanded
- **Direct variable in Python:** `open('$CONFIG_FILE')` is vulnerable; use os.environ
- **Inline python3 -c with interpolation:** `python3 -c "with open('$file')"` is vulnerable

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Shell-to-Python data passing | String interpolation in heredocs | Environment variables | Prevents command injection |
| JSON parsing in bash | Custom awk/sed | Python json module via heredoc | More reliable, handles edge cases |
| Date arithmetic in bash | Manual calculations | Python datetime via heredoc | Handles timezones, ISO8601 properly |

**Key insight:** The complexity of correctly escaping shell variables for Python string literals is too high to be reliable. Environment variables provide a clean boundary that prevents any shell metacharacters from being interpreted.

## Common Pitfalls

### Pitfall 1: Forgetting to Import os
**What goes wrong:** `NameError: name 'os' is not defined`
**Why it happens:** Copy-paste from old code without adding `import os`
**How to avoid:** Always add `import os` when using os.environ
**Warning signs:** Python error in stderr, function returns fallback value

### Pitfall 2: Unquoted Heredoc with Env Vars
**What goes wrong:** Variables are passed via env, but heredoc is unquoted, so $() or backticks in the Python code itself could be expanded
**Why it happens:** Assumption that env vars alone are sufficient
**How to avoid:** Always quote the heredoc delimiter: `<<'EOF'` not `<<EOF`
**Warning signs:** Syntax errors when heredoc contains $-prefixed Python vars

### Pitfall 3: Missing Error Handling for Missing Env Vars
**What goes wrong:** KeyError if environment variable not set
**Why it happens:** Using os.environ['VAR'] when VAR might not exist
**How to avoid:** Use os.environ.get('VAR', 'default') for optional values
**Warning signs:** Python KeyError in stderr

### Pitfall 4: Type Conversion Errors
**What goes wrong:** ValueError when converting non-numeric string to int
**Why it happens:** Assuming shell variable is numeric without validation
**How to avoid:** Use int(os.environ.get('VAR', '0')) with fallback
**Warning signs:** Python ValueError in stderr

### CR-03 Fix: _apply_time_decay (logging.sh:174)
```bash
# Current vulnerable:
python3 <<EOF 2>/dev/null || echo "$base_weight"
# ...
ts = datetime.fromisoformat('$timestamp'.replace('Z', '+00:00'))
# ...
decay_weight = $base_weight * (0.5 ** (age_days / $half_life_days))

# Fixed:
TIMESTAMP="$timestamp" BASE_WEIGHT="$base_weight" HALF_LIFE_DAYS="$half_life_days" python3 <<'EOF' 2>/dev/null || echo "$base_weight"
import os
from datetime import datetime
import math

try:
    timestamp = os.environ['TIMESTAMP']
    base_weight = float(os.environ['BASE_WEIGHT'])
    half_life_days = float(os.environ['HALF_LIFE_DAYS'])

    ts = datetime.fromisoformat(timestamp.replace('Z', '+00:00'))
    now = datetime.now(ts.tzinfo) if ts.tzinfo else datetime.now()
    age_days = (now - ts).total_seconds() / 86400

    decay_weight = base_weight * (0.5 ** (age_days / half_life_days))
    print(decay_weight)
except:
    print(os.environ.get('BASE_WEIGHT', '1.0'))
EOF
```

### CR-05 Fix: read_config (ai-delegate:107)
```bash
# Current vulnerable:
value=$(python3 -c "
import json
try:
    with open('$CONFIG_FILE') as f:
        cfg = json.load(f)
    keys = '$key'.split('.')
    # ...
except: pass
" 2>/dev/null)

# Fixed:
value=$(CONFIG_FILE="$CONFIG_FILE" KEY="$key" python3 <<'EOF' 2>/dev/null
import json
import os

try:
    config_file = os.environ['CONFIG_FILE']
    key = os.environ['KEY']

    with open(config_file) as f:
        cfg = json.load(f)
    keys = key.split('.')
    v = cfg
    for k in keys:
        if isinstance(v, dict):
            v = v.get(k)
        else:
            v = None
            break
    if v is not None:
        if isinstance(v, bool):
            print('true' if v else 'false')
        elif isinstance(v, (list, dict)):
            print(json.dumps(v))
        else:
            print(v)
except:
    pass
EOF
)
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Shell interpolation in heredocs | Env var + quoted heredoc | 2020s security hardening | Eliminates command injection vector |
| `python3 -c "$code"` | `python3 <<'EOF'...EOF` | Recommended for multi-line | Better readability, quoting safety |
| Triple-quote escaping | os.environ | Always | No need to escape special chars |

**Deprecated/outdated:**
- Triple-quoted string interpolation (`'''$var'''`): Still vulnerable to shell expansion before Python sees it
- Unquoted heredocs with careful escaping: Error-prone, always use quoted heredocs
