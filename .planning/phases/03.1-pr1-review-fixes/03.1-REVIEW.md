### CR-01: Command Injection via Python Heredoc with Shell Variable Interpolation

**File:** `/home/zzs/dev/.meta/bin/lib/escalation.sh:353-374`
**Issue:** The `_validate_escalation_chain()` function uses a Python heredoc that interpolates the `$chain_json` variable directly into the script without proper escaping. If `chain_json` contains malicious content (e.g., from a crafted config file), it could execute arbitrary Python code.

### CR-02: Command Injection in log_task_outcome via Triple-Quoted String Interpolation

**File:** `/home/zzs/dev/.meta/bin/lib/logging.sh:89-104`
**Issue:** The `log_task_outcome()` function uses triple-quoted strings (`"""`) to interpolate `$task_type` and `$model` directly into Python code. While less severe than CR-01 (these values are typically controlled), a crafted task_type containing `"""` could escape the string context.

**Fix:**
```bash
# Current (potentially unsafe):
python3 <<EOF >> "$DELEGATION_LOG" 2>/dev/null || true
import json
from datetime import datetime

entry = {
    "timestamp": datetime.now().isoformat(),
    "event_type": "task_outcome",
    "task_type": """$task_type""",  # Unsafe if task_type contains """
    "model": """$model""",
    "success": $py_success,
    "duration_ms": $duration_ms,
    "escalation_count": $escalation_count,
    "tokens_used": $tokens_used
}
print(json.dumps(entry))
EOF

# Secure alternative:
TASK_TYPE="$task_type" MODEL="$model" PY_SUCCESS="$py_success" \
DURATION_MS="$duration_ms" ESCALATION_COUNT="$escalation_count" TOKENS_USED="$tokens_used" \
python3 <<'EOF' >> "$DELEGATION_LOG" 2>/dev/null || true
import json
import os
from datetime import datetime

entry = {
    "timestamp": datetime.now().isoformat(),
    "event_type": "task_outcome",
    "task_type": os.environ['TASK_TYPE'],

    "success": os.environ['PY_SUCCESS'] == 'true',
```

**File:** `/home/zzs/dev/.meta/bin/ai-delegate:399-432`
**Issue:** The `cmd_parallel()` function spawns multiple background jobs that all write to `$DELEGATION_LOG` concurrently. While JSONL append is generally atomic on POSIX filesystems, concurrent writes can interleave bytes, corrupting log entries.

### WR-06: Missing Validation in read_config Fallback

**File:** `/home/zzs/dev/.meta/bin/ai-delegate:82-121`
**Issue:** The `read_config()` function's Python fallback doesn't validate that the returned value is well-formed. If the config file contains non-JSON-serializable types or the JSON is truncated, the function returns partial data without error, leading to silent misconfigurations.

**Fix:**
```bash
read_config() {
    local key="$1"
    local default="$2"

    if [[ ! -f "$CONFIG_FILE" ]]; then
        echo "$default"
        return
    fi

    local value
    if command -v jq &>/dev/null; then
        value=$(jq -r ".${key} // empty" "$CONFIG_FILE" 2>/dev/null)
    else
        value=$(KEY="$key" DEFAULT="$default" python3 -c "
import json
import sys
import os

try:
    with open(os.environ.get('CONFIG_FILE', '.planning/config.json')) as f:
        cfg = json.load(f)
    keys = os.environ['KEY'].split('.')
    v = cfg
    for k in keys:
        if isinstance(v, dict):
            v = v.get(k)
        else:
            v = None
            break
    if v is not None:
        if isinstance(v, bool):
            print('true' if v else 'false')
        elif isinstance(v, (list, dict)):
            print(json.dumps(v))
        else:
            print(v)
    else:
        print(os.environ.get('DEFAULT', ''))
except Exception:
    print(os.environ.get('DEFAULT', ''))
" 2>/dev/null)
    fi

    # Validate result is not empty when a non-empty default was provided
    if [[ -z "$value" && -n "$default" ]]; then
        value="$default"
    fi

    echo "$value"
}
```
