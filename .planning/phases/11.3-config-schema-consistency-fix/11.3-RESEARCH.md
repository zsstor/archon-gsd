# Phase 11.3: Config Schema Consistency Fix - Research

**Researched:** 2026-04-18
**Domain:** Configuration schema / JSON parsing / Bash shell scripting
**Confidence:** HIGH

## Summary

The `task_routing` section in `.planning/config.json` has an inconsistent schema: some entries use array format (e.g., `"scaffold": ["gemini-flash", "glm-ollama"]`) while others use object format with a nested `models` array (e.g., `"impl": { "models": [...], "scoring_mode": "...", "model_constraints": "..." }`). This inconsistency was introduced during Phase 11 when `model_constraints` was added to certain task types, requiring conversion from array to object format.

The problem manifests when `log_routing_decision()` in logging.sh records `model_chain` to the delegation log. The code reads the config value directly via `read_config "task_routing.impl"` which returns the entire object `{"models": [...], "scoring_mode": "...", "model_constraints": "..."}` instead of just the array `[...]`. This causes `model_chain` to be logged incorrectly, breaking downstream analysis that expects an array.

**Primary recommendation:** Normalize all `task_routing` entries to object format with a required `models` array, then update all code that reads `task_routing.<type>` to extract just the `models` field when an array is needed.

## Architecture Patterns

### Pattern 1: Schema Normalization (Recommended)

**What:** Convert all `task_routing` entries to object format with a required `models` array.

**When to use:** When all entries need consistent structure and some entries require additional metadata.

**Example:**
```json
{
  "task_routing": {
    "impl": {
      "models": ["gemini-flash", "glm-ollama", "glm-zai", "claude-sonnet"],
      "scoring_mode": "keyword+files",
      "model_constraints": "tier-2-minimum"
    },
    "scaffold": {
      "models": ["gemini-flash", "glm-ollama"]
    },
    "tdd-cycle": {
      "models": ["codex", "claude-sonnet"]
    },
    "code-review": {
      "models": ["codex", "claude-opus"],
      "model_constraints": "cross-family-preferred"
    },
    "complex-refactor": {
      "models": ["codex", "claude-opus"]
    },
    "judgment": {
      "models": ["claude-opus"],
      "model_constraints": "opus-only"
    },
    "trivial": {
      "models": ["claude-haiku", "gemini-flash"]
    }
  }
}
```

**Pros:**
- Consistent structure everywhere
- Extensible: can add fields to any entry later
- Single code path for reading

**Cons:**
- Config is slightly more verbose
- Requires code changes to read `.models` instead of direct value

### Pattern 2: Polymorphic Reading (Alternative)

**What:** Keep both formats in config, update code to handle both.

**Example code:**
```bash
_get_model_chain() {
    local task_type="$1"
    local raw
    raw=$(read_config "task_routing.${task_type}" "[]")

    # Handle both array and object formats
    if echo "$raw" | jq -e 'type == "array"' > /dev/null 2>&1; then
        echo "$raw"
    elif echo "$raw" | jq -e '.models | type == "array"' > /dev/null 2>&1; then
        echo "$raw" | jq '.models'
    else
        echo "[]"
    fi
}
```

**Pros:**
- No config migration needed
- Backward compatible

**Cons:**
- Every read site needs this logic
- Inconsistent config harder to maintain
- Duplication across multiple files

### Recommendation

**Use Pattern 1 (Schema Normalization).** The config migration is small (4 entries to convert), and consistent structure prevents future bugs. Pattern 2 adds complexity to every read site.

## Common Pitfalls

### Pitfall 1: Incomplete Migration

**What goes wrong:** Updating config but missing some code read sites. Some commands still pass object to `model_chain`, others pass array.

**Why it happens:** The codebase has 6+ places reading `task_routing.<type>`, spread across ai-delegate and routing.sh.

**How to avoid:** Create a helper function `_get_model_chain()` that handles the extraction, call it everywhere.

**Warning signs:** After fix, run `ai-delegate -v impl "test"` and check log entry contains array.

### Pitfall 2: Breaking scoring_mode Reads

**What goes wrong:** While fixing model_chain extraction, accidentally breaking `_get_scoring_mode()` which correctly reads the object format already.

**Why it happens:** Over-eager refactoring of all config reads.

**How to avoid:** `_get_scoring_mode()` in routing.sh:54-73 already handles object format correctly. Leave it alone.

**Warning signs:** `_get_scoring_mode "impl"` should return `"keyword+files"`, not empty.

### Pitfall 3: Fallback Chain Confusion

**What goes wrong:** Forgetting that `read_config` has a fallback default. If extraction fails, returning wrong default.

**Why it happens:** The helper should return `'["gemini-flash"]'` as fallback, not `'[]'` or the entire object.

**How to avoid:** Test with non-existent task type to verify fallback behavior.

## Code Examples

### Helper Function to Extract Model Chain

```bash
# Add to routing.sh (or logging.sh, depending on who calls it most)

# Get model chain array for task type
# Handles both array format (legacy) and object format (with .models)
# Args: task_type
# Returns: JSON array string
_get_model_chain() {
    local task_type="$1"

    if [[ ! -f "$CONFIG_FILE" ]]; then
        echo '["gemini-flash"]'
        return
    fi

    local raw
    if command -v jq &>/dev/null; then
        # Try .models first (object format), fall back to direct value (array format)
        raw=$(jq -r "
            .task_routing.\"${task_type}\" |
            if type == \"object\" then .models else . end //
            .task_routing.impl |
            if type == \"object\" then .models else . end //
            [\"gemini-flash\"]
        " "$CONFIG_FILE" 2>/dev/null)
    else
        raw=$(TASK_TYPE="$task_type" CONFIG_FILE="$CONFIG_FILE" python3 <<'EOF' 2>/dev/null
import json
import os

task_type = os.environ['TASK_TYPE']
config_file = os.environ['CONFIG_FILE']

try:
    with open(config_file) as f:
        cfg = json.load(f)
    routing = cfg.get('task_routing', {})
    entry = routing.get(task_type, routing.get('impl', ['gemini-flash']))

    if isinstance(entry, dict):
        print(json.dumps(entry.get('models', ['gemini-flash'])))
    elif isinstance(entry, list):
        print(json.dumps(entry))
    else:
        print('["gemini-flash"]')
except:
    print('["gemini-flash"]')
EOF
)
    fi

    echo "${raw:-[\"gemini-flash\"]}"
}
```

### Config Migration Script

```python
#!/usr/bin/env python3
# migrate_task_routing.py - Normalize task_routing to object format

import json
import sys

CONFIG_PATH = '.planning/config.json'

with open(CONFIG_PATH, 'r') as f:
    cfg = json.load(f)

routing = cfg.get('task_routing', {})

for task_type, entry in routing.items():
    if isinstance(entry, list):
        # Convert array to object with models key
        routing[task_type] = {'models': entry}
    # Object entries already have models key (or should)

cfg['task_routing'] = routing

with open(CONFIG_PATH, 'w') as f:
    json.dump(cfg, f, indent=2)

print(f"Migrated {len([e for e in routing.values() if isinstance(e, list)])} entries")
```
