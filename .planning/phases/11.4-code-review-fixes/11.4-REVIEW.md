## Critical Issues

### CR-01: Command Injection via Unescaped Task Type in jq

**File:** `/home/zzs/dev/.meta/bin/lib/routing.sh:55`
**Issue:** The `$task_type` variable is interpolated directly into a jq selector without escaping or validation. If `task_type` contains special characters like quotes or backslashes, it could break out of the jq expression and potentially execute arbitrary code or cause parsing errors.

**Fix:**
```bash
# Use jq's --arg flag to safely pass variables
mode=$(jq -r --arg tt "$task_type" '.task_routing[$tt].scoring_mode // "keyword"' "$CONFIG_FILE" 2>/dev/null)
```

### CR-02: Arbitrary Code Execution via .profile Sourcing

**File:** `/home/zzs/dev/.meta/bin/ai-delegate:26`
**Issue:** The script sources `~/.profile` without any validation. A malicious or compromised `.profile` could execute arbitrary commands. This is particularly dangerous since `ai-delegate` may run with elevated privileges or in automated contexts.

**Fix:**
```bash
# Only source if explicitly allowed via environment variable
if [[ "${AI_ALLOW_PROFILE_SOURCE:-false}" == "true" && -f ~/.profile ]]; then
    source ~/.profile 2>/dev/null || true
fi
# Better: require explicit environment variable setting instead
# export GEMINI_API_KEY=... before running ai-delegate
```

### WR-01: Array Access Without Bounds Check

**File:** `/home/zzs/dev/.meta/bin/lib/routing.sh:242-244`
**Issue:** Array elements `${models[1]}` and `${models[2]}` are accessed without checking array length first. If the model chain has fewer than 2 or 3 elements, this will return empty strings, causing the model selection to fail.

**Fix:**
```bash
if [[ $complexity_score -ge 70 && ${#models[@]} -ge 3 ]]; then
    selected_model="${models[2]:-${models[0]}}"
elif [[ $complexity_score -ge 40 && ${#models[@]} -ge 2 ]]; then
    selected_model="${models[1]:-${models[0]}}"
else
    selected_model="${models[0]:-gemini-flash}"
fi
```

### WR-02: Non-Atomic Config File Modification

**File:** `/home/zzs/dev/.meta/bin/lib/routing.sh:358-387`
**Issue:** Config file is modified in-place without atomic write (write to temp file, then move). If the script is interrupted during JSON writing, the config file could be corrupted.

**Fix:**
```bash
python3 <<EOF
import json
import tempfile
import shutil
import os

try:
    with open('$CONFIG_FILE', 'r') as f:
        cfg = json.load(f)

    # ... modification logic ...

    # Atomic write via temp file
    fd, temp_path = tempfile.mkstemp(dir=os.path.dirname('$CONFIG_FILE'), prefix='.config-', suffix='.json')
    try:
        with os.fdopen(fd, 'w') as f:
            json.dump(cfg, f, indent=2)
        shutil.move(temp_path, '$CONFIG_FILE')
    except:
        os.unlink(temp_path)
        raise
except Exception as e:
    pass
EOF
```

### WR-03: Python Dependency Without Fallback

**File:** `/home/zzs/dev/.meta/bin/lib/routing.sh:195,200`
**Issue:** Float comparison relies on `python3` availability. If Python is not installed or fails, the upgrade trigger logic silently fails (returns 1), which could prevent necessary scoring mode upgrades.

**Fix:**

```bash
    should_upgrade=0

    should_upgrade=0
fi
```

### WR-06: File Read Without Existence Check

**File:** `/home/zzs/dev/.meta/bin/ai-delegate:175-180`
**Issue:** File `$PROJECT_ROOT/$context_file` is checked with `-f` but `cat` is executed without checking read permissions. If file exists but isn't readable, `cat` will fail and the error will be embedded in the prompt.

**Fix:**
```bash
if [[ -n "$context_file" && -f "$PROJECT_ROOT/$context_file" && -r "$PROJECT_ROOT/$context_file" ]]; then
    local context_content
    if context_content=$(cat "$PROJECT_ROOT/$context_file" 2>&1); then
        prompt="$description

Context from $context_file:
$context_content

Implement this following the existing patterns in the codebase."
    else
        log "Warning: Could not read context file: $context_file"
        prompt="$description"
    fi
fi
```

### WR-07: Datetime Parsing Without Error Handling

**File:** `/home/zzs/dev/.meta/bin/lib/routing.sh:432-434`
**Issue:** Datetime parsing in the weighted failure rate calculation uses `try/except` in Python, but silently falls back to weight=1.0 on any error. This could mask data quality issues in the delegation log.

**Fix:**
```python
# Add logging for parse failures
try:
    ts = datetime.fromisoformat(ts_str.replace('Z', '+00:00'))
    now = datetime.now(ts.tzinfo) if ts.tzinfo else datetime.now()
    age_days = (now - ts).total_seconds() / 86400
    weight = 0.5 ** (age_days / half_life_days)
except Exception as e:
    # Log parse failure for debugging
    import sys
    print(f"Warning: Failed to parse timestamp '{ts_str}': {e}", file=sys.stderr)
    weight = 1.0
```
