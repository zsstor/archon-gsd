# Phase 11.4: Code Review Fixes - Research

**Researched:** 2026-04-18
**Domain:** Bash shell scripting security and error handling
**Confidence:** HIGH

## Summary

This phase addresses 15 code review findings from Phase 11.3 in two bash scripts: `routing.sh` and `ai-delegate`. The findings span critical security vulnerabilities (command injection, unsafe sourcing), warnings (missing bounds checks, non-atomic writes, missing existence checks), and informational improvements (better error context, constants for magic numbers).

All fixes involve standard bash security patterns and error handling techniques. The research confirms well-documented solutions exist for each finding category. The critical issues (CR-01, CR-02) require immediate attention as they represent actual security vulnerabilities. Warning fixes (WR-01 through WR-08) improve robustness and prevent runtime failures. Info fixes (IN-01 through IN-05) enhance maintainability.

**Primary recommendation:** Apply jq `--arg` flag for CR-01, opt-in environment variable for CR-02, and add defensive checks throughout both files using standard bash idioms.

### Pattern 1: Safe jq Variable Passing
**What:** Use `--arg` flag instead of string interpolation in jq selectors
**When to use:** Any time a shell variable is used in a jq filter
**Example:**
```bash
# Source: https://jqlang.org/manual/
# UNSAFE - CR-01 vulnerability
mode=$(jq -r ".task_routing.\"${task_type}\".scoring_mode" "$CONFIG_FILE")

# SAFE - use --arg to pass variable
mode=$(jq -r --arg tt "$task_type" '.task_routing[$tt].scoring_mode // "keyword"' "$CONFIG_FILE")
```
[VERIFIED: jqlang.org/manual, sqlpey.com/bash/shell-variables-in-jq-filters]

### Pattern 2: Opt-in Profile Sourcing
**What:** Only source profile files when explicitly enabled via environment variable
**When to use:** Scripts that may run in automated/privileged contexts
**Example:**
```bash
# Source: https://developer.apple.com/library/archive/documentation/OpenSource/Conceptual/ShellScripting/ShellScriptSecurity/
# UNSAFE - CR-02 vulnerability
[[ -z "${GEMINI_API_KEY:-}" && -f ~/.profile ]] && source ~/.profile

# SAFE - require explicit opt-in
if [[ "${AI_DELEGATE_SOURCE_PROFILE:-false}" == "true" ]]; then
    [[ -z "${GEMINI_API_KEY:-}" && -f ~/.profile ]] && source ~/.profile 2>/dev/null || true
fi
```
[VERIFIED: developer.apple.com Shell Script Security]

### Pattern 3: Array Bounds Checking
**What:** Check array length before accessing specific indices
**When to use:** Any time accessing array elements beyond index 0
**Example:**
```bash
# Source: https://www.shellscript.sh/examples/
# UNSAFE - WR-01 vulnerability
selected_model="${models[2]}"

# SAFE - check bounds with fallback
if [[ $complexity_score -ge 70 && ${#models[@]} -ge 3 ]]; then
    selected_model="${models[2]:-${models[0]}}"
elif [[ $complexity_score -ge 40 && ${#models[@]} -ge 2 ]]; then
    selected_model="${models[1]:-${models[0]}}"
else
    selected_model="${models[0]:-gemini-flash}"
fi
```
[VERIFIED: shellscript.sh, copyprogramming.com]

### Pattern 4: Atomic File Writes
**What:** Write to temp file first, then move to final destination
**When to use:** Modifying config files or any critical data
**Example:**
```bash
# Source: https://opensource.com/article/22/6/make-temporary-file-bash
# Create temp file in same directory (ensures same filesystem for atomic mv)
temp_file=$(mktemp -p "$(dirname "$CONFIG_FILE")" .config-XXXXXX.json)
trap 'rm -f "$temp_file"' EXIT

# Write to temp file
python3 <<EOF > "$temp_file"
import json
# ... modification logic ...
EOF

# Atomic move
mv "$temp_file" "$CONFIG_FILE"
trap - EXIT  # Clear trap on success
```
[VERIFIED: opensource.com, putorius.net]

### Pattern 5: Module Existence Check
**What:** Verify required modules exist before sourcing
**When to use:** Scripts with modular architecture using `source`
**Example:**
```bash
# Source: https://www.baeldung.com/linux/shell-script-force-source
# Check all modules exist before sourcing any
for module in routing execution logging quota escalation; do
    module_path="${SCRIPT_DIR}/lib/${module}.sh"
    if [[ ! -f "$module_path" ]]; then
        echo "ERROR: Required module not found: ${module}.sh" >&2
        exit 1
    fi
done

# Now source them
source "${SCRIPT_DIR}/lib/routing.sh"
# ... etc
```
[VERIFIED: baeldung.com, redsymbol.net bash strict mode]

### Pattern 6: Float Comparison with awk
**What:** Use awk for float comparison instead of Python
**When to use:** Simple numeric comparisons where Python startup is overkill
**Example:**
```bash
# Source: https://copyprogramming.com/howto/shell-bash-if-compare-float-numbers
# SLOW - Python startup overhead
if python3 -c "exit(0 if $failure_rate > 0.3 else 1)" 2>/dev/null; then

# FAST - awk is always available
if awk -v fr="$failure_rate" 'BEGIN { exit(fr > 0.3 ? 0 : 1) }'; then
    should_upgrade=0
fi
```
[VERIFIED: copyprogramming.com, s0ands0.github.io]

### Pattern 7: File Permission Check
**What:** Check read permission with `-r` before attempting to read
**When to use:** Before `cat`, reading file contents
**Example:**
```bash
# Source: https://www.cyberciti.biz/faq/unix-linux-shell-scripting-test-if-filewritable/
# Check both existence and readability
if [[ -n "$context_file" && -f "$PROJECT_ROOT/$context_file" && -r "$PROJECT_ROOT/$context_file" ]]; then
    local context_content
    if context_content=$(cat "$PROJECT_ROOT/$context_file" 2>&1); then
        prompt="$description\n\nContext from $context_file:\n$context_content"
    else
        log "Warning: Could not read context file: $context_file"
    fi
fi
```
[VERIFIED: cyberciti.biz, labex.io]

### Pattern 8: Enhanced die() Function
**What:** Include file and line number in error messages
**When to use:** Fatal error functions in bash scripts
**Example:**
```bash
# Source: https://opensource.com/article/22/7/print-stack-trace-bash-scripts
die() {
    local caller_file="${BASH_SOURCE[1]##*/}"
    local caller_line="${BASH_LINENO[0]}"
    log "ERROR: $* (${caller_file}:${caller_line})"
    exit 1
}
```
[VERIFIED: opensource.com, bashsupport.com]

### Anti-Patterns to Avoid
- **String interpolation in jq selectors:** Enables command injection [CR-01]
- **Unconditional profile sourcing:** Arbitrary code execution risk [CR-02]
- **Array access without bounds check:** Empty string or error on short arrays [WR-01]
- **Direct file modification:** Risk of corruption on interrupt [WR-02]
- **Python for simple float comparison:** Unnecessary startup overhead [WR-03]
- **Source without existence check:** Cryptic error on missing file [WR-04]
- **File read without permission check:** Silent failures or error in output [WR-06]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Safe jq variable passing | String escaping | `jq --arg` | Built-in, handles all edge cases |
| Atomic file writes | Direct write + sync | mktemp + mv | Atomic on POSIX, handles crashes |
| Float comparison in bash | Bash arithmetic | awk | POSIX standard, no dependencies |
| Temp file creation | Manual naming | mktemp | Avoids race conditions, secure permissions |
| File locking | Manual PID files | flock | Already used in escalation.sh, fd 9 |

**Key insight:** Bash has standard patterns for all these problems. The existing codebase already uses some (flock for locking, python3 for JSON), the issue is inconsistent application.

## Common Pitfalls

### Pitfall 1: jq String Interpolation Injection
**What goes wrong:** Shell variables containing special characters (quotes, backslashes) break out of jq selectors
**Why it happens:** Easy to overlook when variable content is "usually" alphanumeric
**How to avoid:** Always use `--arg` or `--argjson` for shell variables in jq
**Warning signs:** Any `jq -r ".foo.${variable}.bar"` pattern

### Pitfall 2: Sourcing Files Without Validation
**What goes wrong:** Malicious or corrupted file executes arbitrary code
**Why it happens:** Developer assumes file is trusted because it's on local system
**How to avoid:** Require explicit opt-in via environment variable for profile sourcing
**Warning signs:** `source ~/.profile` without conditional check

### Pitfall 3: Array Index Out of Bounds
**What goes wrong:** Accessing `${array[2]}` on 2-element array returns empty string
**Why it happens:** Bash doesn't error on out-of-bounds access
**How to avoid:** Check `${#array[@]}` before accessing specific indices
**Warning signs:** Hardcoded array indices without length check

### Pitfall 4: Non-Atomic Config Updates
**What goes wrong:** Interrupted write leaves corrupted JSON file
**Why it happens:** Direct write to destination file is simpler
**How to avoid:** Write to temp file in same directory, then `mv`
**Warning signs:** `> "$CONFIG_FILE"` or `>>` without temp file intermediary

### Pitfall 5: Python for Simple Operations
**What goes wrong:** ~100ms startup overhead per invocation
**Why it happens:** Python is familiar, already used for complex JSON
**How to avoid:** Use awk for simple numeric operations (sub-millisecond)
**Warning signs:** `python3 -c "exit(0 if $x > Y else 1)"`

### Pitfall 6: Silent Source Failures with set -e
**What goes wrong:** Missing module causes immediate exit with no error message
**Why it happens:** `set -e` exits on any error, including missing source file
**How to avoid:** Check file existence before sourcing, provide clear error message
**Warning signs:** `source "$path"` without prior `-f` check
